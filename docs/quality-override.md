# 品質チェックリスト追加分 — Web API

> Layer-0（共通）+ Layer-1（言語別）のチェックリストに**追加**する項目のみ。

## API固有の品質基準

### エンドポイント設計
- [ ] RESTful 命名規則に従っている（名詞複数形、ケバブケース）
- [ ] HTTPステータスコードが適切（200/201/204/400/401/403/404/409/500）
- [ ] ページネーションが実装されている（リスト系エンドポイント）
- [ ] レスポンスにリクエストIDが含まれている（トレーサビリティ）

### 認証・認可
- [ ] 認証が必要なエンドポイントに認証ミドルウェアが適用されている
- [ ] 認可チェックがリソース単位で行われている
- [ ] トークン検証が毎リクエスト実行される

### 入力バリデーション
- [ ] リクエストボディのバリデーションが存在する
- [ ] パスパラメータ・クエリパラメータの型チェックがある
- [ ] バリデーションエラーの詳細がレスポンスに含まれる

### E2Eテスト
- [ ] Hurl テストが全エンドポイントの正常系をカバー
- [ ] 認証エラー（401/403）のテストがある
- [ ] バリデーションエラー（400）のテストがある
- [ ] gRPC エンドポイントがある場合、grpcurl テストが存在する

### データベース（依存がある場合）
- [ ] docker-compose.yml でDB環境が再現可能
- [ ] マイグレーションファイルが存在する
- [ ] コネクションプールの設定が適切

### ヘルスチェック
- [ ] `/health` または `/healthz` エンドポイントが存在する
- [ ] ヘルスチェックがDB/外部依存の疎通を確認する
- [ ] Readiness と Liveness が分離されている（k8s/Cloud Run向け）

### レート制限・セキュリティ
- [ ] CORS設定が適切（ワイルドカード`*`は本番で不可）
- [ ] レート制限が設定されている（該当する場合）
- [ ] レスポンスヘッダにセキュリティヘッダが含まれる
