# ADR-001: コアデータモデル設計

**ステータス**: 承認
**日付**: 2026-03-29
**決定者**: Ryusei

## コンテキスト

urushi-chronicle は蒔絵・螺鈿の制作工程を記録するアプリケーションであり、以下のデータを扱う:

1. **作品（works）**: 制作物のメタデータ（技法・素材・状態）
2. **制作工程（process_steps）**: 下塗り→中塗り→上塗り→蒔絵→螺鈿の各ステップ
3. **環境データ（environment_readings）**: IoT センサーからの温度・湿度の時系列データ
4. **画像（images）**: マクロ撮影・工程記録・経年変化の写真

漆工芸では「環境条件と仕上がりの因果関係」が重要であり、制作工程と環境データを紐付けて記録・分析する必要がある。

## 決定

### 1. PostgreSQL + TimescaleDB を採用

- 環境データは 5 分間隔で継続的に記録される時系列データ
- TimescaleDB の hypertable により、時間範囲クエリが高速化される
- PostgreSQL の JSONB 型で `materials_used` のような半構造化データを格納
- 通常のリレーショナルデータ（works, process_steps, images）と時系列データを同一 DB で管理

### 2. UUID v4 を主キーに採用

- 将来的な分散システム対応を見据え、衝突リスクの低い UUID v4 を使用
- Go 側では `github.com/google/uuid` パッケージを使用

### 3. 環境データと工程の紐付け

- `environment_readings` テーブルに `work_id` と `process_step_id` の外部キーを設置
- 特定の工程実行中の環境データをフィルタリング可能にする
- 紐付けは任意（ON DELETE SET NULL）で、センサーデータ単体でも記録可能

### 4. カテゴリは CHECK 制約で管理

- 漆工芸の工程カテゴリは安定しており、頻繁な追加は想定しない
- 別テーブルでの正規化よりも CHECK 制約のほうがシンプルで高速

### 5. バリデーションは Domain Model に集約

- Go の domain 層で入力バリデーションを実施（DB 制約は防御的な二重チェック）
- PRD の要件に基づく制約: 工程名 100 文字、温度 -10〜100℃、湿度 0〜100%、画像 10MB

## 検討した代替案

### InfluxDB / 専用時系列 DB

- **却下理由**: リレーショナルデータと時系列データで別 DB を運用する複雑さが MVP に見合わない
- TimescaleDB は PostgreSQL 拡張であり、単一 DB で両方のワークロードを処理可能

### 連番主キー

- **却下理由**: IoT ゲートウェイからのデータ投入時に ID 生成の同期が不要な UUID が適切

### カテゴリのマスターテーブル化

- **却下理由**: 漆工芸の工程は 8 種類で安定しており、JOIN コストに見合わない

## 影響

- マイグレーションツールは `golang-migrate` 互換の up/down SQL 形式
- TimescaleDB 拡張が DB サーバーに必要（GCP Cloud SQL ではサポート対象）
- `materials_used` を JSONB にしたため、スキーマレス部分の型安全性は Go 側で担保

## リスク

- TimescaleDB のバージョンアップに伴う互換性リスク（低: PostgreSQL 拡張として安定）
- JSONB フィールドのクエリパフォーマンス（低: インデックス追加で対応可能）
