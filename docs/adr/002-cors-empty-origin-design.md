# ADR-002: CORS空文字時の挙動設計

**ステータス**: 承認
**日付**: 2026-04-05
**決定者**: Ryusei

## コンテキスト

urushi-chronicle の REST API は `CORS_ORIGINS` 環境変数でCORS許可オリジンを設定する。
本番環境ではリバースプロキシ（Nginx / Cloud Run の Ingress）を前段に配置し、
フロントエンドとAPIを同一オリジンで提供する構成を想定している。

開発環境ではフロントエンド（`localhost:3000` / `localhost:5173`）とバックエンド（`localhost:8080`）が
異なるオリジンで動作するため、CORSヘッダーが必要になる。

## 決定

### `CORS_ORIGINS` が空文字の場合、CORSヘッダーを一切付与しない

- 本番環境: リバースプロキシ後ろで同一オリジン提供のため、CORSヘッダーは不要
- 開発環境: `CORS_ORIGINS=http://localhost:3000,http://localhost:5173` を `.env` で明示設定
- `CORS_ORIGINS=*`（ワイルドカード）は**サポートしない** — 許可リスト方式のみ

### 実装詳細

`pkg/middleware/cors.go` の `CORS()` 関数:
1. `allowedOrigins` が空文字の場合、ミドルウェアをバイパスして `next` をそのまま返す
2. 非空の場合、カンマ区切りでパースしてホワイトリストマップを構築
3. リクエストの `Origin` ヘッダーがマップに存在する場合のみ `Access-Control-Allow-Origin` を返す

## 検討した代替案

### 空文字時にデフォルトで `*` を設定する

- **却下理由**: 設定漏れで本番環境がオープンCORSになるリスク。Fail-closed が安全

### 空文字時にエラーで起動を止める

- **却下理由**: リバースプロキシ構成ではCORS不要であり、必須にすると不必要な設定負担

## 影響

- 開発者は `.env.example` を参照して `CORS_ORIGINS` を設定する必要がある
- 本番環境ではリバースプロキシがCORS処理を担当する場合、`CORS_ORIGINS` を未設定のままデプロイ可能
- フロントエンドを別ドメインでホストする場合は `CORS_ORIGINS` に明示的にドメインを追加する

## リスク

- 設定漏れでフロントエンドからAPIアクセスできない（低: `.env.example` にデフォルト値記載済み）
- リバースプロキシ構成変更時にCORS設定の追加が必要になる可能性（低: 構成変更時に検証すべき項目として認知）
